\section*{Flooding and Ledgers}
In this stage of evolution in our peer-to-peer network we have introduced Flooding and a Ledger system.\\
The design is mainly the same with the exception of the following:
\begin{itemize}
    \item \textbf{Join Protocol:} This protocal is responsible for dynamically connecting all peers in the network when they join. Any joining peers sends a \textsc{GetPeers} message which create a two-way connection with the corresponding peer, which also sends a \textsc{PeerList} message back, which contains its address book for all peers it knows. The joining node then connects to any unknown nodes/address in the list, then floods the network with a \textsc{Join} message, updating all local address books.
    \item \textbf{Flooding:} To dissemniate transactions and any announcements we use dedouplicating for the flooding by assigning every message a global unqiue \textsc{MsgID}. All peers maintian a local \textsc{seenMsgs} map/list, keeping track of recieved messages and forwards them or ignoring any messages it has already seen, preventing infinite loops (broadcast storms)
    \item \textbf{Ledger:} The network now has a ledger containing the state of all accounts. Each peer has a local copy of this ledger which updates when a unique non-duplicate transaction is recieved. This version holds no bias towards overdrafts and executes any transactions delivered even if the accounts goes negative. Because the flooding mechanisme ensure every peer eventually receives every transactions once, the local ledger eventually converges to the same state, hence eventual consistencey. Note that this is not a perfect implementation which is explained later in the report.
\end{itemize}

\section*{Deduplicating Flooding}
We actually partly achieve Eventual Consistency, but only if the network is fully formed before any transaction are made, if a new peer joins after the transactions are made, Eventual Consistency fails hard.\\\\
\noindent The reason it works is because we use the \textsc{seenMsgs} map which is forwarded once to every neighbor. In a fully connected network this guarantees every peer will eventually recieve the transactions and update its ledger, which leads to identical ledgers.\\
But if a peer joins late, after transactions are completed, it will recieve the \textsc{seenMsgs} map but it won't update its ledger, syncing it with the other peers ledger. The new Peers ledger will be empty until further transactions are made, which means eventual consistency will never be achieve after this point.\\
\section*{Negative Account}
We don't believe it would the network would achieve eventual consistency if overdrafts are rejected.\\
In peer-to-peer networks messsages arrive at differetn times and often out of order. If we reject overdrafts the ledgers would ruin any chance of consistency.\\
Imagine this scenario: If we had a network consisting og Søren, Alex and Thor, where Søren had $100 kr.$ in everyones ledgers. If Søren wants to send $100 kr.$ to both Alex and Thor, then peer A and peer B creates a transactiosn each simultaneously. Depending on which peer first receives the transaction we get two scenario where, Thor has $100 kr.$ and Alex has $0 kr.$ OR Thor has $0 kr.$ and Alex has $100 kr.$ in all ledgers.\\
Because peers process messages in differetn order and reject overdrafts, the ledgers will never converge, breaking eventual consistency.
