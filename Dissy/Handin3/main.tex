\section*{To Encrypt and then Sign  or to Sign an then Encrypt...That is the Question}
The task is to argue which protocol doesn't satisfy the following safety policies:
\begin{itemize}
    \item When a request for information arrives, The database D should be able to determine which user sent
the request.
    \item A user should not be able to get get information about which data other users asked for.
\end{itemize}
The protocols that are given in the exercise are:

\begin{align}
&E_{pk_D}(R),\: S_{sk_A}(E_{pk_D}(R)), \: A   &\text{Encrypt then Sign}\\
&E_{pk_D}(R, S_{sk_A}(R)), \: A                &\text{Sign then Encrypt}
\end{align}
It's assumed reader knows the syntax of the protocols.\\\\
\noindent For a fast conclusion look to the protocol (1). This protocols tells the user to encrypt a request and then add their signature before sending it to the Database. It's assumed that attacks on the network traffic is possible, this means a datapack constructed from protocol (1) can be picked up by an attacker, likely another user, and be manipulated.\\
If such an attack occured it would be possible to removed the current signature $S_{sk_A}$ and replace it with a new signature $S_{sk_B}$ and send it to the Database, which would recognise the attacking users signature and return the request $E_{pk_B}$ with the attacking users private key instead of the intended $E_{pk_A}$.\\
This a clear break on the security policies in that the attacking user accessed another users request. Compare to protocol(2) where the signature is encryptd, anoher user can't manipulate the ciphertext in such a way to trick the database without destroying the datapacket.\\\\
\noindent This tells us that in general, when it comes to authenticity and digital signatures, \textit{A digital signature commits the signer to exactly what was signed and nothing else} so in the case of protocol (1) The signature only commits to the ciphertext and not the actual request, which doesn't prove the signer owns/made the encrypted plaintext.\\
So apply Singature to the planetext...always!
